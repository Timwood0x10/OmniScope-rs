# Audit: docs/ffi_accuracy_development_plan.md vs current code

- Date: 2026-06-07
- Repo HEAD: `17bea02` (master) — `feat: add FP suppression rules for runtime-internal and single-language noise`
- Plan file: `docs/ffi_accuracy_development_plan.md` (350 lines)
- Validation reports cross-referenced: `docs/release/ffi_demo_validation.md`, `docs/release/bun_validation.md`

Summary: The plan's architecture, pass list, and most file paths match real code. The metric numbers, however, are stale: the plan's "baseline" (precision 37.1%, recall 54.2%) was already moved past by commits dated 2026-06 — the harness now bakes in `BASELINE_TP=18, FP=12, FN=3` (precision 60%, recall 85.7%) on the regression test, and external validation reports give precision 68% / recall 62% on ffi-demo and 0% precision on the bun_alloc real-world fixture. Several "P0" items are already implemented, several "P1" items are partially implemented but not what the plan describes, and one key piece (the unified `tests/fixtures/ffi_accuracy_expectations.json` file with `boundary_kind` / `language_pair` / `requires_ffi_evidence` metadata) does not exist.

## Highest-priority drifts

1. **Baseline numbers are obsolete.** Plan § 2 says TP=13, FP=22, FN=11 (precision 37.1%). The actual `tests/accuracy_regression.rs:86-91` baseline is now TP=18, FP=12, FN=3 (precision 60%, recall 85.7%). A new contributor reading the plan would think the project hasn't moved, when the in-tree harness already meets or exceeds every short-term target. Worse, the *real* problem is documented elsewhere (precision 0/19 on `bun_alloc.ll` per `docs/release/bun_validation.md:75`) — the plan does not mention the bun blocker at all.

2. **codegraph is referenced as a whole section (§ 4, 28 lines) but does not exist in the code.** No `codegraph` dependency in any `Cargo.toml`, no imports, no slice expansion logic from it. A reader will look for a `codegraph` crate that isn't there. Boundary seeding is instead in `crates/omniscope-pass/src/analysis/boundary_seeds.rs` using name patterns + IR call structure.

3. **The "FFI gate" is much narrower than the plan describes.** Plan § P0.3 says high-confidence candidates "should require at least one strong FFI signal" and others should be "downgraded or suppressed". In reality `issue_verifier.rs:155-182` only gates *leak-type* candidates that are *also* in runtime-internal allocators with runtime-internal callers. DoubleFree, UAF, CrossFamily, CallbackEscape all pass through without FFI evidence requirement. The doc reads as a stronger contract than ships.

4. **`tests/fixtures/ffi_accuracy_expectations.json` does not exist.** Plan § P0.1 promises (optional) per-fixture metadata: `boundary_kind`, `language_pair`, `resource_family`, `requires_ffi_evidence`. The actual expectations are inline Rust structs without any FFI-specific metadata fields. The FFI-vs-resource split is done by issue kind only (`FfiMetrics::is_ffi_issue` at `accuracy_regression.rs:721`), not by per-fixture annotation.

5. **Validation reports contradict the plan's optimism on free/UAF.** Plan § P0.5 success criterion: "Pure internal post-free patterns do not dominate FFI metrics". External validation shows confirmed-severity `DoubleFree` false positives on `c_fft_c_bridge.ll`, `c_merkle_tree.ll`, and `zig_main.ll` are flow-insensitive aggregation errors (`ffi_demo_validation.md:418-431`) — the leading release blocker. Plan does not flag this as the dominant accuracy issue.

## Recommended next actions

### Edits to the plan (to bring it in sync)

- **Rewrite § 2 Current Baseline.** Replace 37.1% / 54.2% numbers with current `accuracy_regression.rs` baseline (TP=18/FP=12/FN=3, precision 60%, recall 85.7%) **and** add a "Real-world validation" subsection citing `ffi_demo_validation.md` (corpus precision 68%, recall 62%) and `bun_validation.md` (bun_alloc precision 0/19). The plan currently hides the real failure mode.
- **Remove or de-scope § 4 codegraph.** No codegraph integration exists or is in flight. Replace with a one-paragraph note pointing at `analysis/boundary_seeds.rs` as the actual seed mechanism, or split codegraph into its own "future work" section.
- **Mark P0.1–P0.6 as Status: Done / Partial / Not Started.** Per the table above. P0.2, P0.3 (data structures), P0.4, P0.6 are done. P0.1 (per-fixture JSON metadata) and P0.5 (bitcast/GEP/load/store alias propagation) are partial. Without status annotations the plan reads as 0% complete.
- **Add a "P0.7 DoubleFree/UAF flow-insensitive aggregation" item.** This is the user-visible #1 false positive class per `ffi_demo_validation.md` and the bun_validation report (`bun_validation.md:62`, `:152`). It is not mentioned in the plan but is the biggest accuracy regression blocker today.
- **Add a "P0.8 Allocator-shaped functions flagged as their own leaks" item.** Documented in both validation reports (`ffi_demo_validation.md:443-449`, `bun_validation.md:71-72`); no plan section currently addresses it.
